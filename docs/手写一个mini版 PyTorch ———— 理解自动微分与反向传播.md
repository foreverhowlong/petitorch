# 手写一个mini版 PyTorch ———— 理解自动微分与反向传播

## 前言
今年寒假的时候，大致了解了下深度学习的原理，对深度学习大概有一个粗浅的直观理解：

1. 前向传播是对输入空间进行变换、折叠、投影等几何操作，以期能够在输出端得到一个能够被线性空间很好分割的数据分布
2. 梯度下降则要切换到loss地形的视角，按照梯度的指引，根据一定的规则（如随机梯度下降，adam等）走向loss更低的地方。不同的下降策略有不同的擅长点，比如adam收敛更快但是可能会比SGD更容易留在局部最小值，等等。针对棘手的loss地形，我们有比如正则化之类的策略，尽可能达到一个接近全局最低的、平缓的（平缓的谷地泛化性好，并且预测具有连续性，不容易受到对抗性攻击）

但是问题在于，我还是不太能理解这个梯度是怎样计算出来的。“反向传播”这个名词倒是听得很熟了，但是它究竟如何传播的？在高中我们就知道了什么是链式法则，那反向传播和链式法则究竟有什么关系和区别呢？

这两个问题大概算是比较有趣的问题。我觉得要解答这个问题的最好办法，就是手写一个mini版的pytorch，我管他叫做petitorch。

## 要写一个petitorch，究竟是要写什么？
pytorch作为一个框架，实现的功能五花八门，我们显然不能以一己之力全部实现。
针对我们要理解反向传播的目标，我们的迷你版pytorch大概需要实现：

为了完成我们的学习目标，我们先省略了对于底层数学函数的编写，算子的设备适配优化，将这些部分作为我们未来的进一步目标。在底层的数学函数上，我们先行使用numpy代替。

## 各模块的关系结构是怎样的？
为了代码清晰易于理解，将代码进行模块化划分是一个很好的做法。pytorch在这方面给了我们很大的启发。

对于神经网络模块，它只关心神经网络的结构。它用来描述一个层的构成，比如Linear层是 xW + b，它并不在乎梯度是如何计算的，也不在乎参数如何被更新，它只是描述一个网络结构。

对于自动微分引擎模块，它所关心的不是高层面的Layer，而是数据和具体的算子的关系。在autograd看来，不存在什么Linear、MSE，它只能看到加法、乘法、幂等在硬件设备看来最为基础的算子。如果一个运算需要计算梯度，那autograd就会接管这个前向传播，并且在前向传播的过程中构建计算图（这是pytorch 动态图的精髓，详见下文），并且在反向传播阶段把梯度沿着这个计算图的反向一层一层传播回去。

对于优化器模块，它所做的事情更为纯粹。它绑定了一系列的参数，并且在你调用step()的时候用这个参数内部保存的梯度信息，依照一定的规则更新这个参数。

既然这三个模块的职责都清楚了，那我们先进入重头戏————自动微分引擎

## 自动微分引擎
我们所好奇的有关“反向传播”的问题，和这个模块的关联最大。要理解反向传播，我们不能不理解“计算图”的概念。
所谓计算图，就是数据在由输入变成输出的过程中，经历的计算步骤的拓扑关系的体现。比如向量X在流经一个线性层的时候，它所构建的计算图就是

``` 
  [ Input: x ]          [ Weights: W ]
       │                      │
       │                      │
       │                      │
       └───────╮      ╭───────┘
               ▼      ▼
            [  MatMul  ] ───► ( x·W )
                  │
                  ▼
            [   Add    ] ◄─── [ Bias: b ]
                  │
                  ▼
             [ Output: y ]
```
一个数据可能是多个数据的父节点，也可能是多个数据的子节点，所以我们必须通过建立图结构来完整地记载这种拓扑关系，从而正确地计算、分配梯度。

建立计算图有两种主流的方法：动态图和静态图。

tensorflow的老版本采取的是静态图，在计算之前确定出完整的图结构，优点是运算速度更快，但不好调试。
pytorch采取的动态图路线则是在前向传播的同时记录节点的前后关系，允许开发者像调试一般程序一样在其中打印调试信息，甚至打断点。

刚刚我们展示的计算图的图式已经能很好地体现计算图的大概样貌了，但是如果要细究，我们需要注意到，计算图实际上是-数据-算子-数据-这种...ABA...结构的二分图。在前向传播时，我们关心权重与数据的流动，把算子视作为边；在反向传播的时候，我们关心的是梯度的流动、如何pull-back梯度，这些都跟算子更有关，所以我们把算子视作为节点。

理解了计算图作为二分图的本质，我们就能理解，为什么算子的闭包（也就是携带一定记忆的算子实例）一定要存它上游的数据，而数据内部也要存它上游的算子。（注：我所说的上游，指的是离输入更近的一段，下游相反）

在这里需要注意的是，我们对计算图的想象不要局限于线性结构。我们在使用pytorch的时候，可能经常使用nn.Sequential，将不同层串联起来；在很多关于神经网络的科普视频中也常常将神经网络展现为串联的形态。这可能导致我们把计算图想象成“输入数据经过一系列通道”的过程。这种心智模型会让我们认为计算图大概是一个近似线性的结构，并且输入数据和权重是完全不同的东西，在反向传播的时候需要分开处理。但是事实上，计算图可以分岔非常多，我们甚至可以把好几个不同结构的网络的结果通过某种方式并在一起；同时，在计算图中，权重和数据没有任何区别，都只是Tensor的一个实例，只是权重需要梯度，输入数据不需要梯度而已。（requires_grad这个属性在下一段有讲解）

***图关系有了，那梯度存到哪里去呢？***

其实这个问题很好回答。我们需要注意到，在优化器更新权重的时候，它是直接从数据中获取到它的梯度，进行更新的。从维度的角度来说，梯度和权重也必然是同维度的。从而，把梯度和数据存在一起，就是很名正言顺的选择了。我们给这样的，存有权重/中间值、梯度、边关系的数据一个名字，叫做Tensor（张量）。

提到Tensor，它其实还储存有一个重要信息：requires_grad。从字面上很好理解，我希望autograd引擎能够计算出这个Tensor的梯度。既然我要这个Tensor的梯度，那它下游的梯度就是必不可少的（这是由链式法则决定的）。从而，这个requires_grad性质就一定是具有传染性的，上游会传染给下游。同时，这个requires_grad也是“把这个运算交给自动微分引擎”的一个信号。如果它并不需要梯度，那前向传播的时候我们就可以不保存数据，也可以少去建图的开销。只有它是需要梯度的，那它下游的所有运算才需要保存数据，建到计算图中去。


***计算图有了，Tensor也有了，那我们究竟应该以什么顺序去算计算图中不同节点的梯度呢？***

我们当然可以从前向传播的上游开始算，先算Loss对第一层权重的梯度，在这一过程中，我们由于复合函数求导法则，必然会计算Loss对最后一层输入的梯度，倒数第二层输出对它的输入的梯度。如果我们接下来继续算Loss对第二层权重的梯度，我们就会发现我们还是要计算Loss对最后一层输入的梯度。这种重复子问题的出现提示我们需要关注计算图的结构。实际上，我们在构建计算图的时候，永远不会吧一个输出喂给上游的计算过程作为输入。计算图是一个有向无环图。有向无环图和重复子问题，很明确地指向动态规划的思想。如果我们从Loss对最后一层输入的梯度算起，一层一层往回、往上游传递，就能避免很多重复的计算。这种将这一层的输出的梯度（也就是这个输出对于L的贡献）定责到每一个该层每一个输入的梯度（输入对于L的贡献）的过程，就是pull-back。**在计算图上从下游回溯到上游的动态规划算法，就是反向传播的本质。**

***问题又来了，pull-back究竟是如何实现的？***

比较一般的讲法似乎不太关心这个问题，大多是通过分量的微分学、维度分析等给出一个答案。我想在这个问题里给出一个有趣一点的视角。但这个是否理解并不影响我们对反向传播的工程实现。

考虑 $n$ 输入、 $m$ 输出的多元向量值函数 $y=f(x)$ ，其中 $x$ 是一个 $n$ 维向量。  
假设这个函数是可微的，它的增量函数存在一个线性近似，这个线性近似是一个线性映射，记为 $J$。  
从而，我们有
$$dy = Jdx$$
这里， $J$ 将数据传往下游，称为pull-forward算子。

考虑到标量损失函数 $L$ 的全微分具有形式不变性，无论是在 $x$ 空间还是 $y$ 空间观察，标量增量 $dL$ 必须相等：
$$\langle \nabla_y L, dy\rangle = \langle \nabla_x L, dx\rangle$$
代入 $dy$ 得
$$\langle \nabla_y L, Jdx\rangle = \langle \nabla_x L, dx\rangle$$
我们定义伴随算子 $J^*$ ，使得
$$\langle \nabla_y L, Jdx\rangle = \langle J^* \nabla_y L, dx\rangle$$
通过对比，我们可以发现
$$J^* \nabla_y L = \nabla_x L$$
通过这个式子，我们就完成了把 $\nabla_y L$ pull-back回 $\nabla_x L$ 的过程。  
特别地，在实数矩阵与标准内积定义中，我们可以发现 $J^* = J^T$ ，这就是全线性层pullback要乘以矩阵转置的原因。在深度学习的高维张量运算中，我们通常不需要显式地构造出庞大的雅可比矩阵，而是直接通过算子的特性，隐式地计算向量与雅可比矩阵的乘积。
这一个思想，就是pytorch中VJP（向量雅克比积）的内容。

从刚刚的推导中，我们可以发现：要pullback梯度，我们在反向传播的时候就必然需要存储一些信息。比如Y=WX + b，如果我们要计算对于X的梯度，就必须储存W的信息。这就对算子提出了要求：它的梯度必须要能够很容易地通过前向传播的时候储存的信息进行计算。如果不能达到这个要求，这个算子往往就不会被神经网络大规模运用。

***这个用来反向传播的信息，应该存在哪里呢？***

显然，这个信息是和算子的类型是紧密相关的，要用这个信息进行梯度pull-back，我们就必须知道这个信息如何嵌入到反向传播的公式中去的，所以它不能存放在Tensor中，而是应该放到算子的闭包中去。

我们总结这个闭包需要管理的信息：首先，它应该绑定一个算子，从而知道正向传播、反向传播的具体计算方法；其次，为了构成二分图，它应该储存一个Tensor的引用，让它知道反向传播计算出的梯度应该给谁；最后，为了能够顺利计算梯度，它还需要储存反向传播需要的信息，比如输入数据，比如输入的维度。

到现在，我们已经清楚了计算图的结构，节点中储存的信息，以及反向传播作为一种从下游向上遍历计算图的一种动态规划算法的本质。我们只有最后一块拼图了：

***如何实现在计算这个梯度的时候，已经获得了所有它下游的梯度？***

假设我们计算梯度是顺序执行的，我们就必须要求在计算一个梯度的时候，所有必须的下游梯度都已经算好了。这是一个排序问题，我们称为**拓扑排序**。

拓扑排序有多种实现方法，我们采用的是深度优先搜索的方法。通过递归调用，先发现自己未访问的前驱的节点先加入列表中，越先加入的，受依赖就越小。Loss节点会被最后加入列表中。得到这个优先列表之后，倒序遍历这个数组计算梯度，就能确保梯度计算顺序的正确性。

至此，我们大致已经了解了autograd引擎的全貌。

## 神经网络模块

这个部分的复杂度相比autograd低很多，它的唯一用处是对于几种运算的封装抽象。比如Linear层，它就是一个矩阵乘法和一个加法。我们在前向传播的时候，实际上还是通过Linear层调用了算子，通过算子动态建立计算图。

同属语法糖的，还有Parameters类。这个类实际上就是强制requires_grad = True的Tensor，强调它是需要通过优化器更新的参数，仅此而已。

一般来说，一个模块会维护和他有关的所有Parameters，并且提供一个方法，可以递归地返回它持有的参数，以及它子模块、子模块的子模块…… 存储的Parameters的引用。

## 优化器

在pytorch的设计中，不同部分的耦合很低，优化器这一部分尤其如此。它在初始化的时候绑定一系列的参数的引用，然后在你调用step()的时候，根据Tensor储存的梯度和优化器对应的规则更新参数。对于他来说，梯度就是“魔法般获得的”。所以按理说，你甚至可以自己手算每一个参数的梯度，然后要求优化器更新。（虽然绝对没有人这么做就是了）
